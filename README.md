# BakeCake: пошаговый план разработки

## 1. Общие правила работы

### Главная цель MVP

Нам нужно быстро собрать рабочий сайт для приёма заказов на торты.

Сайт должен уметь:

- показать лендинг;
- показать стандартные торты;
- дать собрать свой торт;
- оформить заказ;
- сохранить заказ в базе;
- дать пользователю посмотреть свои заказы;
- дать менеджеру увидеть заказ в админке.

### Что сейчас не делаем

На старте не нужно добавлять:

- сложную корзину;
- промокоды;
- сложную аналитику;
- полноценную CRM;
- много разных ролей пользователей;

## 2. Итог, к которому мы должны прийти

К концу MVP должны быть готовы:

- Django-проект с рабочей структурой;
- главная страница на Django templates;
- каталог стандартных тортов;
- конструктор кастомного торта;
- оформление заказа;
- регистрация/вход по телефону и коду;
- личный кабинет;
- список заказов пользователя;
- админка для менеджера;
- сохранение UTM-меток заказа;
- экспорт заказов в CSV.

## 3. Порядок разработки

Разработка делится на 10 этапов.

Важно: не прыгать сразу к сложным вещам. Каждый следующий этап зависит от предыдущего.

---

## Этап 1. Подготовка проекта

### Что делаем

- Инициализируем Git-репозиторий, если он ещё не инициализирован.
- Уточняем и фиксируем `.gitignore`.
- Устанавливаем зависимости из `requirements.txt`.
- Создаём Django-проект.
- Проверяем, что `runserver` стартует без ошибок.

### Что должно появиться

- `manage.py`
- папка `config/`
- рабочие Django settings
- подключённые `templates/` и `static/`

### Результат

- проект запускается;
- открывается тестовая страница Django;
- статика подключается;
- шаблон `templates/index.html` может рендериться через Django.

---

## Этап 2. Базовая структура приложений

### Что мы делаем

Создаём Django apps:

- `core`
- `accounts`
- `catalog`
- `orders`

### Что кладём в каждое приложение

`core`:

- главная страница;
- общие шаблоны;
- legal pages;
- общие утилиты.

`accounts`:

- пользователь;
- OTP-коды;
- авторизация;
- профиль.

`catalog`:

- стандартные торты;
- опции конструктора;
- цены.

`orders`:

- создание заказа;
- статусы;
- расчёт стоимости;
- история заказов;
- экспорт.

### Результат

- все apps добавлены в `INSTALLED_APPS`;
- у каждого app есть `models.py`, `views.py`, `urls.py`, `admin.py`;
- роуты подключаются без конфликтов.

---

## Этап 3. Перенос frontend в Django templates

### Что делаем

- Подключаем шаблоны через Django.
- Делаем базовый шаблон `base.html`.
- Выносим общие части:
  - header;
  - footer;
  - auth modal;
  - общие script/style includes.
- Подключаем текущий CSS и JS через `{% static %}`.

### Что важно

На этом этапе не нужно менять дизайн без причины.

Наша задача:

- сохранить текущую вёрстку;
- сделать её управляемой из Django;
- убрать дублирование.

### Результат

- главная страница визуально осталась похожей на текущую;
- кабинеты и модалки открываются;
- ссылки на картинки, шрифты и JS не сломаны.

---

## Этап 4. Пользователь и авторизация

### Что делаем

- Создаём кастомную модель пользователя.
- Авторизация строится по телефону.
- Добавляем модель OTP-кода.
- Делаем flow:
  - пользователь вводит телефон;
  - получает код;
  - вводит код;
  - система создаёт пользователя или логинит существующего.

### Что пока можно упростить

Для локальной разработки не нужен реальный SMS-провайдер.

Можно:
- генерировать код;
- писать его в лог/консоль;
- проверять flow вручную.

### Результат

- новый пользователь создаётся после подтверждения кода;
- существующий пользователь входит повторно;
- код имеет срок жизни;
- код нельзя использовать бесконечно.

---

## Этап 5. Модели каталога и опций торта

### Что делаем

- Создаём модель `CakeOption`.
- Создаём модель `CatalogCake`.
- Заполняем опции конструктора по ТЗ:
  - уровни;
  - формы;
  - топпинги;
  - ягоды;
  - декор.

### Что важно

Правильные цены указаны в ТЗ.

### Результат

- все опции заведены в БД;
- цены совпадают с ТЗ;
- админка показывает каталог и опции.

---

## Этап 6. Расчет цен

### Что делаем

Пишем отдельный сервис расчёта стоимости заказа.

Он должен:

- принимать выбранные опции;
- проверять обязательные поля;
- считать сумму;
- добавлять `+500` за надпись;
- добавлять `+20%`, если доставка меньше чем через 24 часа.

### Результат

- цена считается правильно для простого заказа;
- цена считается правильно для заказа с надписью;
- срочная доставка даёт +20%;
- ошибочные входные данные не проходят.

---

## Этап 7. Создание заказа

### Что делаем

- Создаём модель `Order`.
- Добавляем форму оформления заказа.
- Backend принимает данные из checkout.
- Backend заново валидирует заказ.
- Backend пересчитывает цену через "Расчет цен".
- Backend сохраняет:
  - состав заказа;
  - контактные данные;
  - адрес и время доставки;
  - согласие с ПДн;
  - UTM-метки;
  - итоговую стоимость.


### Результат

- заказ создаётся;
- пустые обязательные поля ловятся;
- неверные опции не проходят;
- после оформления пользователь получает страницу успеха;
- заказ виден в БД и в админке.

---

## Этап 8. Личный кабинет

### Что делаем

- Делаем страницу профиля.
- Даём пользователю редактировать:
  - имя;
  - email;
  - адрес по умолчанию.
- Делаем список заказов пользователя.
- Делаем страницу или modal с деталями заказа.

### Что показываем в заказе

- номер заказа;
- состав торта;
- цена;
- статус;
- примерное время доставки;
- комментарии по доставке при необходимости.

### Результат

- неавторизованный пользователь не попадает в кабинет;
- авторизованный видит только свои заказы;
- редактирование профиля сохраняет изменения;
- детали заказа подгружаются корректно.

---

## Этап 9. Staff admin и работа менеджера

### Что делаем

Настраиваем Django admin так, чтобы менеджер мог работать без отдельной CRM.

Нужно:

- список заказов;
- фильтры по статусу;
- поиск по телефону/имени/номеру заказа;
- просмотр состава торта;
- просмотр адреса и времени доставки;
- смена статуса;
- редактирование ETA;
- экспорт CSV.

### Какие статусы нужны на MVP

Достаточно простого набора:

- `new`
- `accepted`
- `in_progress`
- `delivering`
- `completed`
- `cancelled`

### Результат

- менеджер видит все новые заказы;
- менеджер может поменять статус;
- CSV выгружается;
- в CSV есть нужные поля.

---

## Этап 10. UTM и минимальная аналитика

### Что делаем

- Захватываем UTM-параметры при первом заходе пользователя.
- Сохраняем их в сессию или cookie.
- При оформлении заказа переносим UTM в сам заказ.

### Какие поля нужны

- `utm_source`
- `utm_medium`
- `utm_campaign`
- `utm_content`
- `utm_term`
- `referrer`
- `landing_path`

### Результат

- если открыть сайт с `?utm_source=yandex`, метка не теряется;
- при создании заказа метка попадает в БД;
- менеджер может увидеть метку в админке или в CSV.

---

## 4. Что можно делать параллельно

### Backend может делать параллельно

- модель пользователя и OTP;
- модели каталога;
- pricing service;
- модель заказа;
- admin;
- UTM middleware.

### Frontend может делать параллельно

- разрезание шаблонов;
- перенос в `base.html`;
- подключение статических файлов;
- адаптация форм под Django;
- подготовка HTMX/AJAX для модалок и профиля.

### Что нельзя делать отдельно без синхронизации

- названия полей форм;
- имена endpoint;
- формат ответа quote endpoint;
- статусы заказа;
- обязательность полей;
- формат телефона.

Эти вещи backend и frontend должны согласовать заранее.

## 5. Что тестировать обязательно

### Авторизация

- вход нового пользователя;
- вход существующего пользователя;
- неверный код;
- просроченный код.

### Конструктор

- все обязательные опции;
- заказ без необязательных полей;
- заказ с надписью;
- заказ со срочной доставкой.

### Заказ

- пустая форма;
- валидная форма;
- неверный телефон;
- неверный email;
- дата/время в прошлом;
- заказ без согласия с ПДн.

### Кабинет

- список заказов;
- детали заказа;
- редактирование профиля;
- защита от доступа к чужому заказу.

### Админка

- изменение статуса;
- фильтры;
- экспорт CSV.

## 6. Минимальные технические соглашения для команды

### По коду

- Каждая новая бизнес-сущность должна иметь модель, admin и базовые тесты.
- Сложная логика должна выноситься в services, а не жить во view.
- Все пользовательские данные валидируются на backend.

### По Git

- Одна задача = одна ветка.
- Один pull request = одна логическая часть работы.
- Не смешивать в одном PR:
  - frontend перенос шаблонов;
  - auth flow;
  - pricing logic;
  - admin export.

## 7. Как понять, что MVP готов

MVP готов, если можно пройти сценарий:

1. Пользователь открывает сайт с рекламной ссылки.
2. Собирает торт или выбирает стандартный.
3. Регистрируется/входит по телефону.
4. Оформляет заказ.
5. Заказ сохраняется в базе с корректной ценой.
6. Менеджер видит заказ в админке.
7. Пользователь видит заказ в личном кабинете.
8. У заказа сохранён рекламный источник.

Если все 8 пунктов работают стабильно, MVP можно считать собранным.

## 8. Что делать после MVP

После MVP можно переходить к улучшениям:

- online payment;
- промокоды;
- повтор заказа;
- жалобы/обратная связь;
- интеграция с CRM;
- PostgreSQL;
- отчёты и dashboard.

Но это только после того, как базовый сценарий заказа работает без сбоев.
